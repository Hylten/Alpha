### **THE "NO-DESIGN" WRITING STYLE (Strictly Text)**

Since the website is minimalist and text-heavy, **Typography is the Design**. The writer must follow these rules for **every blog post and homepage copy**:

#### **The "Cold" Open**
- Start with a blunt, factual statement about the market or problem
- No warmth, no "In today's fast-paced world," no metaphors
- Make the reader immediately understand: This is about data, infrastructure, and competitive advantage

**Good Openings:**
- "Information asymmetry generates 200-300 bps of deal multiple spread between pre-market and auction processes."
- "The average PE firm sources deals 6-9 months after exit signals become visible to the data."
- "Fund III+ operations deploy $500M+ annually without systematic signal infrastructure."

**Bad Openings:**
- "Discover the future of deal sourcing"
- "In today's competitive PE landscape..."
- "Finding the right deal is more important than ever"

#### **The "Heavy" Middle**
- Use technical terminology: EBITDA, LPs, GPs, IRR, Information Arbitrage, Signal Confidence, Portfolio Orchestration
- If a layman understands the article perfectly, it is **too soft**
- Assume the reader is a PE executive with 10+ years experience
- Use data, frameworks, and measurable metrics—not opinions

**Good Middle Section:**
- "CFO changes (0.95 confidence), tech stack upgrades (0.80 confidence), and founder age + no succession (0.85 confidence) cluster with 92% accuracy in 12-month exit windows."
- "Early-stage signal detection compresses the information window from 18-24 months to 6-9 months, enabling entry negotiation before formal advisor engagement."

**Bad Middle Section:**
- "Our team has years of experience"
- "Clients love our approach"
- "We use cutting-edge technology"
- Fluffy adjectives: amazing, incredible, revolutionary, powerful

#### **The "Low-Friction" Conclusion**
- NO sales pitches
- NO calls-to-action ("Schedule a demo," "Learn more," "Contact us")
- End with a statement of fact about Roials Alpha's infrastructure or philosophy
- Optionally: "Details in The Alpha Report" or "Roials Alpha builds this infrastructure"

**Good Conclusions:**
- "This is the infrastructure that turns market inefficiency into systematic alpha."
- "Roials Alpha automates this detection layer. The math is available upon request."
- "The firms that systematize this approach capture the alpha."

**Bad Conclusions:**
- "Ready to transform your deal flow?"
- "Let's work together to solve this challenge"
- "Schedule a call to learn more"

#### **Formatting for SEO**
- Use H2 and H3 tags that **contain the target keywords**
- Keep the prose underneath **tight and dense** (2-3 sentence paragraphs)
- Use bullet points for frameworks, not fluffy benefits
- No section should be more than 200 words

### **WRITER'S CHECKLIST FOR "THE ALPHA STYLE"**

Before submitting ANY article, answer these questions:

1. **VOICE & AUTHORITY**
   - [ ] Does it sound like an internal intelligence memo, not a pitch?
   - [ ] Would a PE principal recognize the terminology?
   - [ ] Is the opening statement blunt and data-driven?

2. **ADJECTIVES & FLUFF**
   - [ ] Are there unnecessary adjectives? (Delete them)
   - [ ] Does every sentence add information?
   - [ ] Would a CFO consider this "actionable"?

3. **BUSINESS MODEL CLARITY**
   - [ ] Does it reinforce "Signal Detection" (not lead gen)?
   - [ ] Does it emphasize "Autonomous Infrastructure" (not AI tools)?
   - [ ] Does it position us as solving "Information Arbitrage" (not generic PE software)?

4. **NO MARKETING LANGUAGE**
   - [ ] Are there any clichés? (Delete them)
   - [ ] Does it sound like a sales email? (Rewrite it)
   - [ ] Is there a hidden sales pitch? (Remove it)

5. **SEO TECHNICAL**
   - [ ] Do H2/H3 headers contain target keywords?
   - [ ] Is the meta description under 160 characters?
   - [ ] Are internal links to related infrastructure concepts?

## 🚀 Publishing Workflow (How Plåtniklas works)

To publish a new article:
1. Create a `.md` file in the appropriate content directory:
   - `content/intelligence/` (Capital & Alpha)
   - `content/insights/` (Pathmaker)
2. Ensure correct YAML frontmatter:
   ```yaml
   ---
   title: "Article Title"
   description: "SEOs description"
   date: "2026-03-03"
   author: "Roials Architect"
   slug: "my-article-slug"
   keywords: "keyword1, keyword2"
   ---
   # Markdown content here...
   ```
3. Commit and push to `main`.
4. **GitHub Actions** will automatically:
   - Run `npm run build`.
   - Execute `node scripts/build-seo.js`.
   - Deploy to GitHub Pages (or Vercel/Netlify for custom domains).

## 🛠️ Internal Mechanics

### 1. Route Hijacking (`App.tsx`)
Because these are primarily single-page apps (SPAs) or use `HashRouter`, traditional SEO-friendly URLs are intercepted at the top of `App.tsx`:
```tsx
const path = window.location.pathname;
if (path.includes('/intelligence')) {
  // Capture slug and render the Intelligence components directly
  // Bypasses the main router to allow clean URLs
}
```

### 2. Static HTML Generation (`scripts/build-seo.js`)
Since search engines prefer static pages, this script runs post-build:
- Reads `dist/index.html`.
- Parses every Markdown file in the content directory.
- Creates physical directories (e.g., `dist/intelligence/slug/index.html`).
- Injects the specific `<title>` and `<meta>` tags for that article.
- Generates `sitemap.xml` and `robots.txt` at the root of `dist/`.

### 3. Browser-Safe Parsing
We avoid `gray-matter` in the React components because it depends on Node's `Buffer`. Instead, we use a lightweight `parseFrontmatter` function defined in the Index/Article components to handle metadata extraction in the client.

## ⚠️ Troubleshooting FAQ

### "Black Screen on Sub-folders"
Usually a routing or asset path issue.
- **Fix**: Ensure the `base` in `vite.config.ts` matches the GitHub repo name (e.g., `/Alpha/`).
- **Fix**: Use relative paths in `App.tsx` hijacking logic (using `.includes()` instead of `.startsWith()` helps with sub-directories).

### "404 on Articles"
- **GitHub Pages Subfolder**: Relative links like `/intelligence/slug` fail because they point to the domain root. Links *must* be prefixed with the repo name (e.g., `/Alpha/intelligence/slug`).
- **Trailing Slashes**: URL paths from GitHub Pages often end in `/`. The slug extraction logic in `App.tsx` now strips these automatically: `.replace(/^\/|\/$/g, '')`.

### "Google not finding pages"
- Verify `https://domain.com/sitemap.xml` exists.
- The `build-seo.js` script automatically updates the sitemap with every build.
- Manually submit the sitemap URL to **Google Search Console** for faster indexing.
